# Bug Report — Partial student update nulls `enrolled` and breaks the detail endpoint

| Field | Value |
|---|---|
| **ID** | BUG-20260627-01 |
| **Date** | 2026-06-27 |
| **Reporter** | automated suite (found while writing tests/api/students.spec.ts) |
| **Module** | Students |
| **Severity** | Major |
| **Found by** | API CRUD test — "updates the student and persists the change" |
| **LEOS version** | 0.2.0 |
| **Layer** | API / DB |
| **Status** | ✅ Fixed 2026-06-27 (server/src/lib.rs) — regression-tested |

## Summary
`POST /students/:id/update` with a partial body (one that omits `enrolled`)
writes `enrolled = NULL`. Afterwards `GET /students/:id` fails and returns
`404 {"error":"student not found"}`, so the student's profile can no longer be
opened even though the row still exists.

## Steps to reproduce
1. Create a student: `POST /students {"first_name":"A","last_name":"B"}` → id `N`.
2. Partial update: `POST /students/N/update {"phone":"+91 90000 99999"}` → `200 {"ok":true}`.
3. Read it back: `GET /students/N`.

## Expected
Step 3 returns `200 {"student": { ... "phone":"+91 90000 99999" ... }}`.

## Actual
Step 3 returns `404 {"error":"student not found"}`. The row is still in
`students`, but the read crashes.

## Root cause
- `student_update` (server/src/lib.rs:1008) only COALESCEs `first_name`,
  `last_name`, `card_uid`. Every other column — including `enrolled` — is set to
  the raw incoming value, which is `NULL` when omitted:
  `enrolled = ?9` where `?9 = v["enrolled"].as_bool().map(|b| b as i64)` → `None`.
- `student_detail` (server/src/lib.rs:1072) reads `enrolled` as a non-nullable
  `i64`: `"enrolled": r.get::<_, i64>(9)? == 1`. On a `NULL` it errors, the
  `query_row` returns `Err`, and the handler reports `404`.

## Impact
The desktop UI always sends the full form, so it doesn't trigger this today. But
any partial update (a future PATCH-style call, an integration, or a field-level
save) silently corrupts the row and locks its profile. Two latent defects:
1. Partial updates overwrite unspecified columns with NULL.
2. `student_detail` cannot tolerate a NULL `enrolled`.

## Suggested fix
- Read `enrolled` defensively in `student_detail`:
  `r.get::<_, Option<i64>>(9)?.unwrap_or(0) == 1`.
- And/or COALESCE the preserved columns in `student_update` (mirror the
  first_name/last_name treatment) so omitted fields are left unchanged.

## Resolution (2026-06-27)
Both defects fixed in `server/src/lib.rs`:
1. `student_update` now uses `enrolled=COALESCE(?9,enrolled)` — an omitted
   `enrolled` leaves the stored value unchanged instead of nulling it.
2. `student_detail` now reads `enrolled` as `Option<i64>` and defaults to `0`,
   so a NULL can never crash the read.

Regression test added in `tests/api/students.spec.ts` ("a partial update
preserves untouched fields and still reads back"): a `{ phone }`-only update now
returns 200, persists the phone, and preserves `enrolled = true`. Pre-fix this
returned 404. Note: a partial update still nulls *other* unspecified optional
columns (email/gender/etc.) — that's existing by-design behaviour since the UI
always posts the full form; revisit if a true PATCH endpoint is ever added.
